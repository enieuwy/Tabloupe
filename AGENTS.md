# Agent Knowledge Base: Focus Tab Groups

## Architecture
- **Daemon:** `mac-command-centre` (Swift) listens for macOS Focus mode changes via `DistributedNotificationCenter`, reads the active ID from `Assertions.json`, maps it to a tab group string, and broadcasts it over a `ws://127.0.0.1:8767` WebSocket.
- **Client:** `focus-tab-groups` (Firefox Extension) maintains a persistent WebSocket connection and calls the `tabGroups` API to collapse/expand groups matching the string.
- **Bidirectional bus:** The WebSocket carries two flows. Daemon‚Üíclient: enveloped state multiplexed on one socket ‚Äî Focus (`{ type:"focus", payload:{ focus: null | {id,name,icon,color} } }`), a one-time `focusCatalog` table (`{ type:"focusCatalog", payload:{ entries } }`), plus other subsystems (bluetooth/audio/network/wireguard) the client ignores. Client‚Üídaemon: AI tab-grouping requests (`{ "type": "groupTabs", ‚Ä¶ }`) answered with `{ "type": "groupTabsResult", ‚Ä¶ }`. `WebSocketServer` was broadcast-only until this; it now also `receiveMessage`s and routes inbound frames via `requestHandler`. (WireGuard/Audio commands still arrive via the file-drop control directory, not the socket ‚Äî a browser can't write there.)

## Focus Mode Mappings
- **Catalog source of truth = the daemon.** MCC owns the id ‚Üí `{name, icon, color}` table and broadcasts it as a `focusCatalog` envelope once at startup; `WebSocketServer` replays the latest-per-type snapshot to every client on connect, so a fresh client gets it immediately. Focus envelopes are enriched (`payload.focus` = `null` or `{id,name,icon,color}`). `background.js` caches both into `storage.local.focusCatalog`; `options.js` `focusCatalogEntry()` prefers the cache and falls back to a small bundled table only until the broadcast arrives. Icons travel as **semantic names** (`briefcase`, `moon`, ‚Ä¶) and map to `FOCUS_ICON_PATHS` (SVG paths stay extension-side ‚Äî SF Symbol glyphs don't render reliably in Firefox). `background.js` still accepts a legacy bare-id string for an older daemon.
- **Badge & notifications name the Focus**, not the matched group title ‚Äî `focusDisplayName(rawId)` reads the cached catalog (`Focus: Work`, badge initial `W`).
- **Mapping model is many:many:** `focusMappings[id]` is a `string[]` of tab-group titles (`[]` = seen-but-ignored), and the same group may appear under multiple modes ‚Äî it expands whenever any of them is active. The Options table renders one row per Focus id (icon + name, raw id on `title` hover), removable group **chips**, and a per-row "+ add" input whose autocomplete suggests every tab group except those already in that row (so groups used by other modes are reusable).
- **Unassigned row + drag-and-drop:** The mappings table's last row is "Unassigned" (Firefox tab groups in no mapping). Chips are draggable ‚Äî drop on a Focus row to move it there, **Alt/‚å•-drop to copy** (assign to a second mode without removing the original), drop on the Unassigned row to remove it from its source. Model op: `moveTitleToFocus(title, fromId, toId)` (fromId `null` = copy/add without removing); `makeDropTarget` forwards the drop event so the row chooses copy vs move via `event.altKey`.

## AI Tab Grouping
- **What:** Clusters the current window's ungrouped tabs into named topic groups (like Safari 27's automatic tab groups). Opt-in, preview-before-apply ‚Äî never silent, never touches existing manual groups or pinned/privileged (`about:`, `moz-extension:`) tabs.
- **Where the AI runs:** On-device in the daemon via Apple's FoundationModels framework (`TabGrouping/TabClusterer.swift`, `FoundationModelsTabClusterer`) ‚Äî the same on-device model class behind Safari's feature. Nothing leaves the Mac. Requires macOS 26+; gated with `#if canImport(FoundationModels)` + `@available(macOS 26.0, *)` so the daemon still builds/runs on macOS 14. `SystemLanguageModel.default.availability` is checked per request; when unavailable the client gets `ok:false` + a stable `error` code (e.g. `apple_intelligence_disabled`).
- **Contract:** Client sends `[{ index, title, url }]`; daemon returns `[{ topic, tabIndices }]`. The model only proposes topics + membership ‚Äî the client owns presentation (assigns tab-group colors round-robin) and re-maps indices to concrete tab ids. Both `TabGroupingSanitizer` (daemon) and `mapProposalToGroups` (client) defensively drop out-of-range/duplicate indices and empty groups.
- **Apply path:** `background.js` `applyTabGrouping` calls `browser.tabs.group({tabIds})` then `browser.tabGroups.update(groupId, {title, color})` (Firefox 139+). Generated groups are ordinary tab groups, so they participate in Focus collapse/expand automatically.
- **UX:** The toolbar button (`browser_action` ‚Üí `popup.html`/`popup.js`) is the primary surface. The popup resolves its own window via `windows.getCurrent()`, passes that `windowId` to the background, and on open **auto-previews** when enabled + ‚â•2 groupable tabs + no fresh cache (Apply stays manual: Apply/Regroup/Dismiss). The background caches the last proposal **per window** (`lastProposalByWindow`, 5-min TTL) so the ephemeral popup survives being closed mid-compute and shows results instantly on reopen; the cache is cleared on apply, `ai-group-clear` (Dismiss), and `windows.onRemoved`. Messages: `ai-group-state` ‚Üí `{enabled, groupableCount, proposal}`, `ai-group-preview`, `ai-group-apply`, `ai-group-clear` (all carry `windowId`; it falls back to `WINDOW_ID_CURRENT`). The enable toggle (`aiGroupingEnabled` in `storage.local`) lives in the popup; the Options page only links to it. No new manifest permissions beyond `default_popup`: reuses `tabs`/`tabGroups` and the already-allowed `ws://127.0.0.1:8767`.

## Tab Search
- **What:** Chrome/Edge-style "Search Tabs" overlay ‚Äî a fuzzy-filterable list of every open tab; Enter switches, ‚úï closes a tab. Requested as a Ctrl+S binding.
- **Trigger:** `tabsearch.js` (content script on `<all_urls>`, top frame only) captures the **configured shortcut** in the capture phase and `preventDefault()`s it (default **Ctrl+S**). On macOS Ctrl+S is unbound (Save is Cmd+S) so nothing is overridden; on Win/Linux it shadows "Save Page As" *on web pages only*. A second `commands` entry (`search-tabs`, no default key ‚Äî remappable in `about:addons`) is relayed by the background as `{type:"tabsearch-open"}` so a custom shortcut can also reach privileged-page-adjacent pages. The WebExtension `commands` API itself **cannot** bind Ctrl+S (browser-reserved ‚Üí handler never fires); content-script interception is the only way.
- **Configurable shortcut:** The Options page ("Tab search shortcut" card) records a combo and stores it in `storage.local` as `tabSearchShortcut` = `{ctrl,alt,shift,meta,key}` (or `null` to disable the in-page trigger; absent = default Ctrl+S). `options.js` and `tabsearch.js` each keep an identical `normalizeShortcut` (keep them in sync). `tabsearch.js` reads the value on init and live-updates via `storage.onChanged`; `matchesShortcut` requires an **exact** modifier match so Ctrl+S won't fire while a Ctrl+Shift+S binding is set. The `about:addons` command is independent of this setting.
- **Why a content script:** Content scripts can't call `browser.tabs`, and the overlay can't render in browser chrome. So the UI lives in a **Shadow DOM** root (style-isolated both ways) injected into the page, and all tab data/actions go through the background over messages: `tabsearch-list` ‚Üí `[{id,windowId,title,url,favIconUrl,active,currentWindow}]` (current window floats first, stable order), `tabsearch-activate {tabId,windowId}` ‚Üí `windows.update(focused)` then `tabs.update(active)`, `tabsearch-close {tabId}` ‚Üí `tabs.remove` then returns the refreshed list.
- **Limitation:** No content script runs on privileged pages (`about:`, `addons`, view-source, PDF viewer), so neither Ctrl+S nor the relay can open the overlay there; the command path notifies instead of failing silently.

## Extension Deployment & Updates
- **Standard Firefox enforces extension signing.** You cannot simply load an unsigned local extension permanently unless you use Developer Edition/Nightly.
- **Signing:** Use `web-ext sign --channel="unlisted"` with Mozilla Add-ons (AMO) API keys. The keys are stored in `.env`.
- **Automated Build:** Run `./build.sh`. It automatically bumps the patch version in `manifest.json` (required by AMO), signs the code, and drops a new `.xpi` in `web-ext-artifacts/`.
- **Manual Installation:** After building, you *must* manually go to `about:addons` ‚û°Ô∏è ‚öôÔ∏è ‚û°Ô∏è "Install Add-on From File..." and select the new `.xpi`. Firefox will replace the old version.

## Key Gotchas & Limitations
- **`Assertions.json` Fragility:** Apple provides no public API for reading the *name* of the current Focus mode. We parse `~/Library/DoNotDisturb/DB/Assertions.json`. The schema for this JSON file varies wildly across macOS versions. See `FocusObserver.swift` in `mac-command-centre` for the three fallback key-paths required to extract the ID.
- **Browser Suspension:** Firefox may put the extension's background event page to sleep. `background.js` handles WebSocket disconnects by running an exponential backoff reconnect loop (with a `browser.alarms` fallback).
- **Instant State Recovery:** On a fresh WebSocket connection, the Swift daemon instantly pushes the *current* state. This ensures a restarting browser doesn't have to wait for the *next* focus change to sync.
- **Port Conflicts:** The WebSocket runs on `8767`. In the extension manifest, this requires `"host_permissions": ["ws://127.0.0.1/*"]`.

## Eliminated Cruft
- **Hammerspoon:** Removed `focus_filter.lua`. Detection logic moved to `mac-command-centre`.
- **Rust Host / Unix Sockets / Native Messaging:** Completely bypassed. A direct WebSocket from Swift to Firefox is lighter, requires fewer permissions, and creates a generic event bus for future local apps.
