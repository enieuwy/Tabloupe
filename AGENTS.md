# Agent Knowledge Base: Tabloupe

## Architecture
- **Product:** Tabloupe is a Firefox tab-group view switcher. A **lens** is a saved view of Firefox tab groups; switching one expands matching groups and collapses the rest. Tabs stay open.
- **Client:** `tabloupe` (Firefox MV2, min 139) owns the lens model in `storage.local`, resolves selectors against current `tabGroups`, and uses the Firefox `tabGroups` API to collapse/expand groups. Ungrouped and pinned tabs are never hidden. Only normal windows participate.
- **Automation (optional):** `mac-command-centre` (Swift) can trigger lens activation from macOS Focus. Manual switching works on every OS with no helper, daemon, or WebSocket connection.
- **Bidirectional bus:** The optional `ws://127.0.0.1:8767` WebSocket still carries local automation and AI traffic. Daemon→client frames are enveloped state (`{ type:"focus", payload:{ focus:null | {id,name,icon,color} } }`), `focusCatalog`, and unrelated subsystems the extension ignores. Client→daemon AI frames are `{ "type":"groupTabs", … }` with `{ "type":"groupTabsResult", … }` replies. WireGuard/Audio commands still arrive via the daemon's file-drop control directory, not the socket — a browser cannot write there.

## Lenses
- **Storage model (`schemaVersion: 2`):** `lenses[]` holds `{ id:"lens_<rand>", name, icon, color:null|string, groupSelectors:[{type:"title",value}|{type:"glob",value}], triggers:{ appleFocusIds:string[] }, createdAt, updatedAt, migratedFrom?:{focusIds:[]} }`. `activeView` is persisted globally as `{kind:"all"}` or `{kind:"lens",lensId}`. `lastActivation` is `{trigger:"manual"|"appleFocus"|"schedule"|"external", triggerId?, at}`. `automationFallback` defaults to `{kind:"all"}`. `lensSchedules[]` stores opt-in per-lens schedules `{lensId, enabled, days, start, end}`. `focusSessionHistory[]` stores capped completed sessions with start/end, trigger, view, and expanded/collapsed group snapshots. `ignoredAppleFocusIds` stores seen-but-unbound Apple Focus ids. `legacyFocusMappingsBackup` keeps the old mapping object read-only for one release.
- **Transient and per-window views:** Single-group "Show just this" views are not persisted in `storage.local`. The background holds `Map<windowId, {kind:"transient", label, selectors, windowId}>`; `lens-state` resolves the active view for the requesting window by checking that map before the global `activeView`. The popup's "This window automation" selector stores a `windowProfile` override: follow automation, ignore automation, or always use a specific lens for that window. Both maps are mirrored to `browser.storage.session` (`windowViewState` key) so they survive event-page suspension but are cleared on browser exit (windowIds are only stable within one session); where `storage.session` is missing they degrade to memory-only. Overrides are cleared when the window closes and affect automation/schedule activations, not manual lens switches.
- **Migration:** `migrateToLensesV2()` is gated and idempotent: it runs only when `focusMappings` exists and `schemaVersion !== 2`, writes `schemaVersion:2`, backs up to `legacyFocusMappingsBackup`, and stops writing the old key. Non-empty `focusMappings[id]` become lenses whose selectors come from normalized title/glob entries and whose names/icons/colors come from `focusCatalog` when available. Identical normalized selector sets merge into one lens with multiple `triggers.appleFocusIds`; overlapping-but-different sets stay separate. Empty arrays become `ignoredAppleFocusIds`; catalog-only ids do not create lenses. Fresh installs get zero bundled lenses.
- **Activation:** `activateView(view,{trigger,triggerId})` resolves `{kind:"all"}` to all groups, `{kind:"lens"}` to that lens's `groupSelectors`, and `{kind:"transient"}` to its supplied selectors. It reuses `activateMatchingGroups` for expand/collapse and the active-tab handoff Firefox requires before collapsing the active group. `all` and `lens` persist `activeView` + `lastActivation`; `transient` updates only the per-window map. Manual activation is silent and wins over automation. Automation/schedule activations honor per-window profile overrides before applying group collapse state. The badge shows the active lens initial and clears for All groups.
- **Apple-Focus and schedule adapters:** A Focus envelope id activates the lens whose `triggers.appleFocusIds` contains it. The last automation-applied id is persisted as `lastAppliedAppleFocusId` in `storage.local`, so both guards survive event-page suspension and browser restarts: a same-id reconnect replay is a no-op when `lastActivation.trigger === "manual"` (reconnects do not override a manual switch), and Focus-off applies `automationFallback` only when the current `activeView` was last activated by that same `triggerId`. Unbound ids keep the existing unmapped notification/open-options path and can be bound in Options. The `lens-schedule-tick` alarm exists only while at least one schedule is enabled (`syncScheduleAlarm`, run at startup and on `lensSchedules` changes). The tick is edge-triggered: a matching schedule activates its lens only when `lastActivation.at` predates the current schedule window's start (`scheduleWindowStartMs`, overnight-aware), so it fires once on entering the window and any manual/automation switch after that wins for the rest of the window instead of being yanked back every minute (which also destroyed session-history durations).
- **Messages:** Popup/Options send `lens-state {windowId}` → `{ activeView, windowProfile, lastActivation, lenses:[{id,name,icon,color,active}], currentGroups:[{title,color,savedIn:string[]}], hasGroups, hasAppleBinding, aiEnabled }`; `lens-activate {windowId, view}` where `view` is `{kind:"all"}` | `{kind:"lens",lensId}` | `{kind:"transient",label,selectors:[...]}`; `lens-save {windowId, source:"window"|"group"|"empty", groupTitle?, name, icon?, color?}`; `lens-update {lensId, patch}`; `lens-delete {lensId}`; `lens-link-focus {lensId?, focusId}`; `lens-reorder {orderedIds}`; `window-profile-set {windowId, profile}`. Existing `ai-group-*` and `tabsearch-*` messages remain; `apply-current-focus` is removed.
- **Popup UI:** The toolbar popup is a switcher headed **Tabloupe**. It shows `Showing: <name>` or `Showing: All groups`, `Show all groups`, the current window's automation override selector, saved lenses, this window's groups, `Show just this`, `Save this group`, `Save current groups as a lens`, and `Organize tabs…`. It no longer warns about the daemon in the primary view and no longer auto-previews AI on open; AI lives behind `Organize tabs…`.
- **Options UI:** Options owns lens editing, schedules, import/export, activity history, and optional automation. Lens cards edit name/icon/color and selector chips (`title` or `glob`) under "Matching groups", with unmatched, multi-match, and duplicate-title warnings. Advanced lens controls include per-lens schedule settings. Backup/export excludes the AI provider entirely (no endpoint, no API key); import validates known settings before writing storage and never touches `aiProvider` — a backup containing one surfaces an explanatory toast instead. The provider is only configurable via "Save AI settings" (`saveProvider`), which requires an `https://` base URL (plain `http://` allowed for loopback hosts only — localhost/127.0.0.1/[::1], e.g. Ollama) and the `permissions.request` host-consent prompt; manifest `optional_permissions` and CSP `connect-src` include `http://localhost/*` and `http://127.0.0.1/*` for this. Activity lists recent completed sessions from `focusSessionHistory`. The Automation card is the only primary UI that says **Apple Focus**; it reports helper status and binds detected Apple Focus modes to lenses. Manual switching needs no daemon.
- **AI integration:** "Add new groups to current lens" appends `{type:"title",value:topic}` selectors to the active lens after successful AI apply. The stored key remains `aiPinToFocus` for compatibility, but behavior targets `activeView.kind === "lens"`; with All groups or a transient view active the checkbox is disabled with a hint.

## AI Tab Grouping
- **What:** Clusters the current window's ungrouped tabs into named topic groups (like Safari 27's automatic tab groups). Opt-in, preview-before-apply — never silent from the popup, never touches existing manual groups or pinned/privileged (`about:`, `moz-extension:`) tabs.
- **Where the AI runs:** On-device in the optional daemon via Apple's FoundationModels framework (`TabGrouping/TabClusterer.swift`, `FoundationModelsTabClusterer`) — the same on-device model class behind Safari's feature. Nothing leaves the Mac. Requires macOS 26+; gated with `#if canImport(FoundationModels)` + `@available(macOS 26.0, *)` so the daemon still builds/runs on macOS 14. `SystemLanguageModel.default.availability` is checked per request; when unavailable the client gets `ok:false` + a stable `error` code (e.g. `apple_intelligence_disabled`).
- **Contract:** Client sends `[{ index, title, url }]`; daemon returns `[{ topic, tabIndices }]`. The model only proposes topics + membership — the client owns presentation (assigns tab-group colors round-robin) and re-maps indices to concrete tab ids. Both `TabGroupingSanitizer` (daemon) and `mapProposalToGroups` (client) defensively drop out-of-range/duplicate indices and empty groups.
- **Apply path:** `background.js` `applyTabGrouping(groups, windowId)` calls `browser.tabs.group({tabIds})` then `browser.tabGroups.update(groupId, {title, color})` (Firefox 139+). If a tab group with the proposed topic's title already exists in that window (manual or a prior AI run) it merges into it via `browser.tabs.group({groupId, tabIds})` instead of creating a duplicate (existing title/color preserved). Generated groups are ordinary tab groups, so they participate in lens switching automatically. On a partial failure `commitTabGroups` still pins the successfully applied topics (when pin-to-lens is on) and drops the window's cached proposal — only the failed groups surface as an error.
- **UX:** The popup resolves its own window via `windows.getCurrent()`, passes that `windowId` to the background, and only enters AI grouping from `Organize tabs…`. Opening the popup does not auto-preview. Apply stays manual: Apply/Regroup/Dismiss. The background caches the last proposal per window (`lastProposalByWindow`, 5-min TTL) so the ephemeral popup survives being closed mid-compute and shows results instantly on reopen; the cache is cleared on apply, `ai-group-clear` (Dismiss), and `windows.onRemoved`. Messages: `ai-group-state` → `{enabled, groupableCount, proposal, pinToFocus, activeFocus, autoGroup}`, `ai-group-preview`, `ai-group-apply`, `ai-group-clear` (all carry `windowId`; it falls back to `WINDOW_ID_CURRENT`). The enable toggle (`aiGroupingEnabled` in `storage.local`) lives in the popup; Options only links to it. No new manifest permissions beyond `default_popup`: reuses `tabs`/`tabGroups` and the already-allowed `ws://127.0.0.1:8767`.
- **Grouping prompt (`inscribi-3zrk`):** Options → AI tab grouping shows the full system prompt in a textarea (`#ai-grouping-prompt`), preloaded with the built-in default (`DEFAULT_GROUPING_PROMPT` in `options.js`, kept identical to `GROUPING_SYSTEM_PROMPT` in `background.js`). The user edits it to replace the prompt wholesale (not append), stored as `aiGroupingPrompt` in `storage.local` (trimmed, clamped to 4000 chars via `normalizeGroupingPrompt`). A single "Save AI settings" button (`#provider-save` → `saveProvider`) persists provider config and prompt; a field left equal to the default is stored as an empty override so it keeps tracking future default changes. "Reset prompt to default" (`#ai-prompt-reset` → `resetGroupingPrompt`) clears the override and refills the field with the default. At grouping time `computeTabGrouping` resolves the override (`getGroupingPromptOverride` → `buildGroupingSystemPrompt` = override-or-default): the Custom API path sends it as the system message (`cloudCluster`); the Foundation path forwards it as a `prompt` field on the `groupTabs` frame (`requestTabGrouping`). The daemon ignores `prompt` until MCC adds support, so on-device grouping uses its own prompt for now. The cloud user message (`buildGroupingPrompt`) treats tab titles as attacker-influenced: whitespace/newlines are collapsed and titles capped at 200 chars (`sanitizeTabTitle`) so a page title can't fake extra prompt lines, and a titleless tab falls back to its host only — never the raw URL, which can embed query-string secrets (sign-in/OAuth links) that must not reach a provider's logs.
- **Pin to active lens (opt-in):** A popup checkbox (`aiPinToFocus` in `storage.local`) makes `handleGroupApply` append successfully-applied topics to the active lens's `groupSelectors` (`{type:"title",value:topic}`), so newly created AI groups are included next time that lens is shown. `ai-group-state.activeFocus` reports the active lens name for existing popup copy/disable logic. No lens active means no-op and disabled UI.
- **Continuous auto-grouping (opt-in):** A popup checkbox (`aiAutoGroup` in `storage.local`, shown only when AI grouping is enabled; the popup reads its checked state straight from storage so a stale background that doesn't report `autoGroup` can't silently uncheck it) makes the background silently cluster new ungrouped web tabs. `tabs.onCreated`/`onUpdated(status:"complete")` feed a per-window 5s debounce (`scheduleAutoGroup`) → `runAutoGroup`, gated by an in-memory `autoGroupEnabled` mirror plus two serialization guards: a 30s per-window cooldown (anti-storm) and a per-window in-flight set `autoGroupInflight` (true single-flight). It runs whenever enabled, regardless of active lens, and applies silently via `commitTabGroups(windowId, groups, { notify:false, surfaceEmptyError:false, surfaceStatus:false, pinToActiveLens:false })`: no notification, badge overwrite, `lastError`, or pin-to-lens write. Flipping the toggle on schedules an immediate run for current windows. No new permissions.

## Tab Search
- **What:** Chrome/Edge-style "Search Tabs" overlay — a fuzzy-filterable list of every open tab. Single tab: Enter/click switches, ✕ closes. Multi-select unlocks bulk actions (group, AI-group, close, move-to-new-window, ungroup, pin/unpin, discard). Requested as a Ctrl+S binding.
- **Trigger:** `tabsearch.js` (content script on `<all_urls>`, top frame only) captures the configured shortcut in the capture phase and `preventDefault()`s it (default Ctrl+S); synthetic (non-`isTrusted`) key events are ignored so a page can't open the overlay programmatically. On macOS Ctrl+S is unbound (Save is Cmd+S) so nothing is overridden; on Win/Linux it shadows "Save Page As" on web pages only. The `search-tabs` command also defaults to Ctrl+S (`MacCtrl+S` on macOS) so the background can catch the shortcut on Firefox new-tab/home pages where no content script runs; the fallback navigates that blank Firefox page to the extension-hosted Tab Search page and opens the overlay. The command remains independently remappable in `about:addons`.
- **Configurable shortcut:** The Options page ("Tab search shortcut" card) records a combo and stores it in `storage.local` as `tabSearchShortcut` = `{ctrl,alt,shift,meta,key}` (or `null` to disable the in-page trigger; absent = default Ctrl+S). `options.js` and `tabsearch.js` each keep an identical `normalizeShortcut` (keep them in sync). `tabsearch.js` reads the value on init and live-updates via `storage.onChanged`; `matchesShortcut` requires an exact modifier match so Ctrl+S won't fire while a Ctrl+Shift+S binding is set. The `about:addons` command is independent of this setting.
- **Why a content script:** Content scripts can't call `browser.tabs`, and the overlay can't render in browser chrome. So the UI lives in a **closed** Shadow DOM root (style-isolated both ways; combined with the `isTrusted` gate, the host page can neither open the overlay nor read the rendered tab list out of it — only the module's own `shadow` var reaches the overlay DOM) injected into the page, and all tab data/actions go through the background over messages. The tab list (`tabsearch-list`) is `[{id,windowId,title,url,favIconUrl,active,currentWindow,groupTitle,groupColor,pinned,grouped}]` (current window floats first, stable order). Single-tab: `tabsearch-activate {tabId,windowId}` → `windows.update(focused)` then `tabs.update(active)`; `tabsearch-close {tabId}` → `tabs.remove` then refreshed list. Bulk (all return the refreshed list): `tabsearch-close-many {tabIds}`, `tabsearch-group {tabIds,title,windowId}` (reuses `applyTabGrouping`, so an existing same-title group is merged into), `tabsearch-ungroup {tabIds}`, `tabsearch-move-new-window {tabIds}`, `tabsearch-set-pinned {tabIds,pinned}`, `tabsearch-discard {tabIds}`; `tabsearch-ai-preview {windowId,tabIds}` returns proposed groups and Apply reuses `ai-group-apply`. All bulk handlers live in `background.js`.
- **Web/history fallback rows:** When the query is non-empty, the overlay appends trailing action rows under labeled section headers (`Open tabs` / `History` / `Search`, reusing the `.divider.section-header` style; the per-window divider is suppressed while sectioned): up to 3 `{kind:"history"}` rows then always a final `{kind:"web",query,url}` row. History comes from a debounced (180ms) `tabsearch-history {query}` → `{ok,results:[{title,url}]}` lookup (`browser.history.search({text,maxResults:20,startTime:0})`); the background collapses near-duplicate URLs via `normalizeHistoryUrl` (drops scheme, `www.`, trailing slash, and tracking/session params like `sei`/`utm_*`/`themeRefresh`), keeping the best-titled/shortest, and returns `title:""` for title-less entries. The client drops history whose URL is already an open tab, and requires the query be ≥2 chars before showing history at all (the web row still appears for 1-char queries). History rows use a clock icon (`ICON_CLOCK`), web uses `ICON_GLOBE`; a real title renders title + url subtitle, a title-less entry renders a single readable `host+path` label via `readableUrlLabel` (never the raw URL twice). **URL detection:** `parseQueryAsUrl(query)` resolves the query to a URL when it has an explicit scheme or is an unambiguous host (`localhost[:port]`, an IPv4, `host:port`, or `host.<known-tld>[/…]` against a conservative `KNOWN_TLDS` set — a dotted query with an unknown TLD like `test.foo` or a plain word stays a search). Only `http(s)` schemes are treated as navigable: any other explicit scheme (`javascript:`, `file:`, `data:`, `moz-extension:`, …) falls back to a web search. When set, `row.url` flips the web row into a "Go to `<host+path>`" direct-open row. `filtered[]` holds tab objects (no `kind`) and these `kind`-tagged action rows together, so arrow-nav/Enter span both; action rows carry no checkbox and are skipped by all marking paths (`toggleMark`/`markRange`/Ctrl+Enter). **Shift+Enter** activates the web/go row from any selection (so the user never arrows past tabs/history to reach it). The footer hint verb is contextual to the selected row (`switch` for a tab; `open in new tab` for history/go-to-URL; `search in new tab` for web) plus a matching `⇧↵` hint when a non-web row is selected. `activate(item)` branches on `kind`: web with a `url` → `tabsearch-open-url {url}` else → `tabsearch-web-search {query}` (`browser.search.query({query,disposition:"NEW_TAB"})`); history → `tabsearch-open-url {url}`; tab → existing `tabsearch-activate`. `openUrlFromSearch` opens the URL in a new tab (`browser.tabs.create`) **unless** the current tab is blank (`isBlankTab`: `about:blank`/`about:newtab`/`about:home`/empty), in which case it navigates that tab in place so no empty tab is left behind. Browse mode (empty query) renders no action rows. Adds `history` + `search` to manifest `permissions`.
- **Focus-steal guard:** Because the search input is in the Shadow DOM, key events that bubble out to the page's `document` are retargeted to the host `<div>` (not a form field). Page-level hotkey handlers can react to every keystroke and yank focus to their own search box. `buildOverlay` therefore attaches bubble-phase `keydown`/`keypress`/`keyup` listeners on the host that `stopPropagation()`, so the page never sees keystrokes while the overlay is open. Our own handlers sit on `inputEl` (deeper in the tree) and fire first, so typing/navigation still work. This only blocks bubble-phase page handlers; capture-phase ones are unblockable without breaking our own typing.
- **Reclaiming focus from the address bar:** Navigating the Firefox new-tab page in place (`tabs.update`) leaves keyboard focus stuck in the address bar (Firefox bug 1411465 / 1415860), so typed text lands in the URL bar and the content-side `inputEl.focus()` can't win. `openTabSearchOverlay` therefore opens `tabsearch.html?tabsearchOpen=1` in a **fresh** `tabs.create({active:true})` tab (focus lands in content like a link click) and discards the blank new-tab with `tabs.remove`, rather than navigating it. As a secondary guard, `openOverlay` calls `focusInput()` (`window.focus()` + `inputEl.focus()` retried across ~10 animation frames until `shadow.activeElement === inputEl`), which also covers the command-relay path; harmless on the normal content-script path.
- **Multi-select & bulk actions (mob7):** `marked` is a `Set` of tab ids (survives filtering). Mark via the per-row checkbox, ⌘/Ctrl-click, Shift-click (range), or ⌘/Ctrl+Enter (toggle highlighted + advance); plain click/Enter still activates. Two-stage Esc clears marks if any, else closes. A footer action bar (shown when `marked.size>0`) offers Group / AI group / Close / More (Move to new window · Ungroup · Pin/Unpin · Discard) / Clear. A tab group is window-scoped, so Group/AI-group/Ungroup are disabled when marks span >1 window (hint shown); Close/Move/Pin/Discard work across windows. The list shows faint per-window dividers. Manual Group reveals an inline name field with a `<datalist>` of existing group titles (pick one → merge). AI group swaps the list for a preview of topic sections with Apply/Cancel.
- **Group sections & drag-to-group (browse mode):** With an empty query the list renders group sections — a collapsible header (color badge + member count + chevron, with hover-revealed Ungroup / Close-group buttons) above its indented members (the per-row group pill is suppressed); ungrouped tabs stay flat. Typing falls back to the flat ranked list with pills, since matches scatter across groups. Collapse state lives in `collapsedGroups` (keyed by groupId); a collapsed group's members leave the navigable `filtered` set. Drag a row (or the marked selection, if the dragged row is marked) onto a section header to add those tabs to that group: `tabsearch-group {tabIds, groupId, windowId}` → `browser.tabs.group({groupId, tabIds})` (merge by id, existing title/color kept). The tab-list item therefore also carries `groupId`. Firefox keeps grouped tabs contiguous, so `listTabsForSearch`'s native window+index order already places members adjacent — sectioning is a pure rendering layer.
- **Positional drag (browse mode):** Beyond dropping on a header, a row is a drop target with before/after halves (pointer y vs row mid). A single shared `.drop-indicator` element previews the insertion point. Dropping resolves an anchor tab + `placeAfter` and a target group from the visible neighbours — inside a group only when both neighbours share that real group, otherwise no group — and sends `tabsearch-move {tabIds, windowId, anchorId, placeAfter, groupId}`. The background `moveTabsFromSearch` re-queries the window's live order, computes the destination index, `browser.tabs.move`s there, then explicitly reconciles membership (`tabs.group({groupId})` to join, `tabs.ungroup` to leave).
- **Limitation:** No content script runs on privileged pages (`about:`, `addons`, view-source, PDF viewer). The command fallback handles Firefox new-tab/home pages by replacing that blank page with `tabsearch.html?tabsearchOpen=1`; other privileged pages still notify instead of failing silently. On that standalone page, `closeOverlay` also closes its own tab (`isStandalonePage()` → `window.close()`, with a `browser.tabs.getCurrent`/`remove` fallback), so dismissing the overlay or activating a result never orphans a blank extension tab.

## Extension Deployment & Updates
- **Name:** The add-on is **Tabloupe** (a tab + loupe portmanteau; formerly Tab Lens, originally Focus Tab Groups). `manifest.name`, `browser_action.default_title`, popup heading, and primary copy use Tabloupe; the saved-view object is still a **lens**. Manifest description: "Save and switch views of your Firefox tab groups. Collapse the groups you don't need; tabs stay open. Optional macOS Focus automation."
- **Standard Firefox enforces extension signing.** You cannot simply load an unsigned local extension permanently unless you use Developer Edition/Nightly.
- **Signing:** Use `web-ext sign --channel="unlisted"` with Mozilla Add-ons (AMO) API keys. The keys are stored in `.env`.
- **Automated Build:** Run `npm ci` once, then `./build.sh`. It automatically bumps the patch version in `manifest.json` (required by AMO), signs the code, and drops a new `.xpi` in `web-ext-artifacts/`. Both `build.sh` and `deploy.sh` invoke the lockfile-pinned web-ext via `npx --no-install` (never `npx -y`, which would run an unpinned registry package while AMO signing secrets are in the environment) and fail fast with a "run `npm ci` first" message when it is missing.
- **Deploy script state:** `deploy.sh` keeps its Firefox session snapshots and the lz4 venv under `$HOME/.local/state/tabloupe` (chmod 700), not world-readable `/tmp`; lz4 is version-pinned via `LZ4_VERSION` in its config block.
- **Manual Installation:** After building, you must manually go to `about:addons` → ⚙️ → "Install Add-on From File..." and select the new `.xpi`. Firefox will replace the old version.

## Key Gotchas & Limitations
- **Manual-first:** Switching lenses by hand needs no daemon, no WebSocket, and no macOS APIs. Optional Apple Focus automation only activates lenses that have explicit Apple Focus bindings.
- **`Assertions.json` fragility:** Apple provides no public API for reading the name/id of the current Focus mode. MCC parses `~/Library/DoNotDisturb/DB/Assertions.json`; the schema varies across macOS versions. See `FocusObserver.swift` for the fallback key paths required to extract the ID.
- **Browser suspension:** Firefox may put the non-persistent MV2 background event page to sleep. `background.js` starts from script evaluation on wake, keeps WebSocket reconnect alarm-backed (exponential backoff capped at 10 minutes so an offline daemon doesn't wake the page every minute forever), stores durable state in `storage.local` (including `lastAppliedAppleFocusId`), mirrors per-window view state to `storage.session`, and treats only the proposal cache and in-flight guards as ephemeral.
- **Instant automation recovery:** On a fresh WebSocket connection, the Swift daemon pushes the current Focus state. The adapter must apply the reconnect-replay guard so this does not override a manual lens switch with a stale same-id envelope.
- **Port conflicts:** The optional WebSocket runs on `8767`. In the extension manifest, this requires `"host_permissions": ["ws://127.0.0.1/*"]`.

## Eliminated Cruft
- **Hammerspoon:** Removed `focus_filter.lua`. Detection logic moved to `mac-command-centre` and is now only an optional automation trigger.
- **Rust Host / Unix Sockets / Native Messaging:** Completely bypassed. A direct WebSocket from Swift to Firefox is lighter, requires fewer permissions, and creates a generic event bus for future local apps.
