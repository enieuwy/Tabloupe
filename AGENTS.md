# Agent Knowledge Base: Focus Tab Groups

## Architecture
- **Daemon:** `mac-command-centre` (Swift) listens for macOS Focus mode changes via `DistributedNotificationCenter`, reads the active ID from `Assertions.json`, maps it to a tab group string, and broadcasts it over a `ws://127.0.0.1:8767` WebSocket.
- **Client:** `focus-tab-groups` (Firefox Extension) maintains a persistent WebSocket connection and calls the `tabGroups` API to collapse/expand groups matching the string.
- **Bidirectional bus:** The WebSocket carries two flows. Daemon→client: enveloped state multiplexed on one socket — Focus (`{ type:"focus", payload:{ focus: null | {id,name,icon,color} } }`), a one-time `focusCatalog` table (`{ type:"focusCatalog", payload:{ entries } }`), plus other subsystems (bluetooth/audio/network/wireguard) the client ignores. Client→daemon: AI tab-grouping requests (`{ "type": "groupTabs", … }`) answered with `{ "type": "groupTabsResult", … }`. `WebSocketServer` was broadcast-only until this; it now also `receiveMessage`s and routes inbound frames via `requestHandler`. (WireGuard/Audio commands still arrive via the file-drop control directory, not the socket — a browser can't write there.)

## Focus Mode Mappings
- **Catalog source of truth = the daemon.** MCC owns the id → `{name, icon, color}` table and broadcasts it as a `focusCatalog` envelope once at startup; `WebSocketServer` replays the latest-per-type snapshot to every client on connect, so a fresh client gets it immediately. Focus envelopes are enriched (`payload.focus` = `null` or `{id,name,icon,color}`). `background.js` caches both into `storage.local.focusCatalog`; `options.js` `focusCatalogEntry()` prefers the cache and falls back to a small bundled table only until the broadcast arrives. Icons travel as **semantic names** (`briefcase`, `moon`, …) and map to `FOCUS_ICON_PATHS` (SVG paths stay extension-side — SF Symbol glyphs don't render reliably in Firefox). `handleMessage` requires `payload.focus` to be `null` or an object with a string `id`; any other shape is treated as Focus-off.
- **Badge & notifications name the Focus**, not the matched group title — `focusDisplayName(rawId)` reads the cached catalog (`Focus: Work`, badge initial `W`).
- **Mapping model is many:many:** `focusMappings[id]` is a `string[]` of tab-group titles (`[]` = seen-but-ignored), and the same group may appear under multiple modes — it expands whenever any of them is active. The Options table renders one row per Focus id (icon + name, raw id on `title` hover), removable group **chips**, and a per-row "+ add" input whose autocomplete suggests every tab group except those already in that row (so groups used by other modes are reusable).
- **Glob patterns (advanced):** A `focusMappings[id]` entry containing `*` or `?` is a **glob** (`*` = any run of characters, `?` = exactly one), matched against tab-group titles by `buildTitleMatcher` in `background.js` (plain entries still match exactly, so `Work` ≠ `Workspace` but `Work*` catches both). In the Options table they live behind a per-row **"more options" chevron icon-button** (`.icon-btn-options`, beside the delete button) that reveals a `.row-options` panel; the chevron is tinted (`.has-content`) when the row holds patterns. They are stored as ordinary strings in the same array — `normalizeTitles`/save treat them identically (`addPatternToFocus`, `makePatternChip`, `expandedPatternRows`).
- **Unassigned row + drag-and-drop:** The mappings table's last row is "Unassigned" (Firefox tab groups in no mapping). Chips are draggable — drop on a Focus row to move it there, **Alt/⌥-drop to copy** (assign to a second mode without removing the original), drop on the Unassigned row to remove it from its source. Model op: `moveTitleToFocus(title, fromId, toId)` (fromId `null` = copy/add without removing); `makeDropTarget` forwards the drop event so the row chooses copy vs move via `event.altKey`.

## AI Tab Grouping
- **What:** Clusters the current window's ungrouped tabs into named topic groups (like Safari 27's automatic tab groups). Opt-in, preview-before-apply — never silent, never touches existing manual groups or pinned/privileged (`about:`, `moz-extension:`) tabs.
- **Where the AI runs:** On-device in the daemon via Apple's FoundationModels framework (`TabGrouping/TabClusterer.swift`, `FoundationModelsTabClusterer`) — the same on-device model class behind Safari's feature. Nothing leaves the Mac. Requires macOS 26+; gated with `#if canImport(FoundationModels)` + `@available(macOS 26.0, *)` so the daemon still builds/runs on macOS 14. `SystemLanguageModel.default.availability` is checked per request; when unavailable the client gets `ok:false` + a stable `error` code (e.g. `apple_intelligence_disabled`).
- **Contract:** Client sends `[{ index, title, url }]`; daemon returns `[{ topic, tabIndices }]`. The model only proposes topics + membership — the client owns presentation (assigns tab-group colors round-robin) and re-maps indices to concrete tab ids. Both `TabGroupingSanitizer` (daemon) and `mapProposalToGroups` (client) defensively drop out-of-range/duplicate indices and empty groups.
- **Apply path:** `background.js` `applyTabGrouping(groups, windowId)` calls `browser.tabs.group({tabIds})` then `browser.tabGroups.update(groupId, {title, color})` (Firefox 139+). If a tab group with the proposed topic's title already exists in that window (manual or a prior AI run) it **merges** into it via `browser.tabs.group({groupId, tabIds})` instead of creating a duplicate (existing title/color preserved). Generated groups are ordinary tab groups, so they participate in Focus collapse/expand automatically.
- **UX:** The toolbar button (`browser_action` → `popup.html`/`popup.js`) is the primary surface. The popup resolves its own window via `windows.getCurrent()`, passes that `windowId` to the background, and on open **auto-previews** when enabled + ≥2 groupable tabs + no fresh cache (Apply stays manual: Apply/Regroup/Dismiss). The background caches the last proposal **per window** (`lastProposalByWindow`, 5-min TTL) so the ephemeral popup survives being closed mid-compute and shows results instantly on reopen; the cache is cleared on apply, `ai-group-clear` (Dismiss), and `windows.onRemoved`. Messages: `ai-group-state` → `{enabled, groupableCount, proposal}`, `ai-group-preview`, `ai-group-apply`, `ai-group-clear` (all carry `windowId`; it falls back to `WINDOW_ID_CURRENT`). The enable toggle (`aiGroupingEnabled` in `storage.local`) lives in the popup; the Options page only links to it. No new manifest permissions beyond `default_popup`: reuses `tabs`/`tabGroups` and the already-allowed `ws://127.0.0.1:8767`.
- **Grouping prompt (`inscribi-3zrk`):** Options → AI tab grouping shows the **full system prompt** in a textarea (`#ai-grouping-prompt`), preloaded with the built-in default (`DEFAULT_GROUPING_PROMPT` in `options.js`, kept identical to `GROUPING_SYSTEM_PROMPT` in `background.js`). The user edits it to **replace** the prompt wholesale (not append), stored as `aiGroupingPrompt` in `storage.local` (trimmed, clamped to 4000 chars via `normalizeGroupingPrompt`). A single **"Save AI settings"** button (`#provider-save` → `saveProvider`) persists the provider config **and** the prompt; a field left equal to the default is stored as an **empty override** so it keeps tracking future default changes. **"Reset prompt to default"** (`#ai-prompt-reset` → `resetGroupingPrompt`) clears the override and refills the field with the default. At grouping time `computeTabGrouping` resolves the override (`getGroupingPromptOverride` → `buildGroupingSystemPrompt` = override-or-default): the **Custom API** path sends it as the system message (`cloudCluster`); the **Foundation** path forwards it as a `prompt` field on the `groupTabs` frame (`requestTabGrouping`). The daemon ignores `prompt` until MCC adds support, so on-device grouping uses its own prompt for now.
- **Pin to Focus (opt-in):** A popup checkbox (`aiPinToFocus` in `storage.local`) makes `handleGroupApply` append successfully-applied topics to the **active** Focus's `focusMappings` entry (`pinTopicsToActiveFocus`; a no-op when Focus is off or the option is unchecked) so newly created AI groups aren't collapsed on the next Focus change. `ai-group-state` also returns `pinToFocus` + `activeFocus` (the active Focus's display name) so the popup labels/disables the checkbox accordingly.

## Tab Search
- **What:** Chrome/Edge-style "Search Tabs" overlay — a fuzzy-filterable list of every open tab. Single tab: Enter/click switches, ✕ closes. **Multi-select** unlocks bulk actions (group, AI-group, close, move-to-new-window, ungroup, pin/unpin, discard). Requested as a Ctrl+S binding.
- **Trigger:** `tabsearch.js` (content script on `<all_urls>`, top frame only) captures the **configured shortcut** in the capture phase and `preventDefault()`s it (default **Ctrl+S**). On macOS Ctrl+S is unbound (Save is Cmd+S) so nothing is overridden; on Win/Linux it shadows "Save Page As" *on web pages only*. A second `commands` entry (`search-tabs`, no default key — remappable in `about:addons`) is relayed by the background as `{type:"tabsearch-open"}` so a custom shortcut can also reach privileged-page-adjacent pages. The WebExtension `commands` API itself **cannot** bind Ctrl+S (browser-reserved → handler never fires); content-script interception is the only way.
- **Configurable shortcut:** The Options page ("Tab search shortcut" card) records a combo and stores it in `storage.local` as `tabSearchShortcut` = `{ctrl,alt,shift,meta,key}` (or `null` to disable the in-page trigger; absent = default Ctrl+S). `options.js` and `tabsearch.js` each keep an identical `normalizeShortcut` (keep them in sync). `tabsearch.js` reads the value on init and live-updates via `storage.onChanged`; `matchesShortcut` requires an **exact** modifier match so Ctrl+S won't fire while a Ctrl+Shift+S binding is set. The `about:addons` command is independent of this setting.
- **Why a content script:** Content scripts can't call `browser.tabs`, and the overlay can't render in browser chrome. So the UI lives in a **Shadow DOM** root (style-isolated both ways) injected into the page, and all tab data/actions go through the background over messages. The tab list (`tabsearch-list`) is `[{id,windowId,title,url,favIconUrl,active,currentWindow,groupTitle,groupColor,pinned,grouped}]` (current window floats first, stable order). Single-tab: `tabsearch-activate {tabId,windowId}` → `windows.update(focused)` then `tabs.update(active)`; `tabsearch-close {tabId}` → `tabs.remove` then refreshed list. Bulk (all return the refreshed list): `tabsearch-close-many {tabIds}`, `tabsearch-group {tabIds,title,windowId}` (reuses `applyTabGrouping`, so an existing same-title group is **merged** into), `tabsearch-ungroup {tabIds}`, `tabsearch-move-new-window {tabIds}`, `tabsearch-set-pinned {tabIds,pinned}`, `tabsearch-discard {tabIds}`; `tabsearch-ai-preview {windowId,tabIds}` returns proposed groups and **Apply reuses the existing `ai-group-apply`** (preview-before-apply). All bulk handlers live in `background.js`.
- **Multi-select & bulk actions (mob7):** `marked` is a `Set` of tab **ids** (survives filtering). Mark via the per-row checkbox, **⌘/Ctrl-click**, **Shift-click** (range), or **⌘/Ctrl+Enter** (toggle highlighted + advance); plain click/Enter still *activate*. **Two-stage Esc** clears marks if any, else closes. A footer action bar (shown when `marked.size>0`) offers Group / AI group / Close / More (Move to new window · Ungroup · Pin/Unpin · Discard) / Clear. A tab group is **window-scoped**, so Group/AI-group/Ungroup are disabled when marks span >1 window (hint shown); Close/Move/Pin/Discard work across windows. The list shows faint per-window dividers. Manual **Group** reveals an inline name field with a `<datalist>` of existing group titles (pick one → merge). **AI group** swaps the list for a preview of topic sections with Apply/Cancel.
- **Group sections & drag-to-group (browse mode):** With an **empty query** the list renders **group sections** — a collapsible header (color badge + member count + chevron, with hover-revealed Ungroup / Close-group buttons) above its indented members (the per-row group pill is suppressed); ungrouped tabs stay flat. **Typing** falls back to the flat ranked list with pills, since matches scatter across groups. Collapse state lives in `collapsedGroups` (keyed by groupId); a collapsed group's members leave the navigable `filtered` set. **Drag** a row (or the marked selection, if the dragged row is marked) onto a section header to add those tabs to that group: `tabsearch-group {tabIds, groupId, windowId}` → `browser.tabs.group({groupId, tabIds})` (merge by id, existing title/color kept). The tab-list item therefore also carries `groupId`. Firefox keeps grouped tabs contiguous (`tabs.group`: "All tabs in a tab group must be adjacent"), so `listTabsForSearch`'s native window+index order already places members adjacent — sectioning is a pure rendering layer.
- **Positional drag (browse mode):** Beyond dropping on a header, a row is a drop target with **before/after halves** (pointer y vs row mid). A single shared `.drop-indicator` element — a **flat full-width line in the gap** between rows (positioned by the reference row's `offsetTop`/`offsetHeight`, so before-next and after-prev resolve to the same y and crossing the threshold doesn't flicker) — previews the insertion point. Dropping resolves an anchor tab + `placeAfter` and a target group from the visible neighbours — **inside a group only when both neighbours share that real group**, otherwise no group — and sends `tabsearch-move {tabIds, windowId, anchorId, placeAfter, groupId}`. The background `moveTabsFromSearch` re-queries the window's live order, computes the destination index (discounting moved tabs already before it), `browser.tabs.move`s there, then **explicitly reconciles membership** (`tabs.group({groupId})` to join, `tabs.ungroup` to leave) rather than relying on implicit move-time grouping. So you can drop a tab **between** two specific tabs and **drag a tab out** of a group by dropping it among ungrouped rows. Positional drops are gated to browse mode (the flat search order isn't the strip order).
- **Limitation:** No content script runs on privileged pages (`about:`, `addons`, view-source, PDF viewer), so neither Ctrl+S nor the relay can open the overlay there; the command path notifies instead of failing silently.

## Extension Deployment & Updates
- **Standard Firefox enforces extension signing.** You cannot simply load an unsigned local extension permanently unless you use Developer Edition/Nightly.
- **Signing:** Use `web-ext sign --channel="unlisted"` with Mozilla Add-ons (AMO) API keys. The keys are stored in `.env`.
- **Automated Build:** Run `./build.sh`. It automatically bumps the patch version in `manifest.json` (required by AMO), signs the code, and drops a new `.xpi` in `web-ext-artifacts/`.
- **Manual Installation:** After building, you *must* manually go to `about:addons` ➡️ ⚙️ ➡️ "Install Add-on From File..." and select the new `.xpi`. Firefox will replace the old version.

## Key Gotchas & Limitations
- **`Assertions.json` Fragility:** Apple provides no public API for reading the *name* of the current Focus mode. We parse `~/Library/DoNotDisturb/DB/Assertions.json`. The schema for this JSON file varies wildly across macOS versions. See `FocusObserver.swift` in `mac-command-centre` for the three fallback key-paths required to extract the ID.
- **Browser Suspension:** Firefox may put the extension's background event page to sleep. `background.js` handles WebSocket disconnects by running an exponential backoff reconnect loop (with a `browser.alarms` fallback).
- **Instant State Recovery:** On a fresh WebSocket connection, the Swift daemon instantly pushes the *current* state. This ensures a restarting browser doesn't have to wait for the *next* focus change to sync.
- **Port Conflicts:** The WebSocket runs on `8767`. In the extension manifest, this requires `"host_permissions": ["ws://127.0.0.1/*"]`.

## Eliminated Cruft
- **Hammerspoon:** Removed `focus_filter.lua`. Detection logic moved to `mac-command-centre`.
- **Rust Host / Unix Sockets / Native Messaging:** Completely bypassed. A direct WebSocket from Swift to Firefox is lighter, requires fewer permissions, and creates a generic event bus for future local apps.
